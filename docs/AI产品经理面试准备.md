# 第一部分：AI 产品经理面试准备

## 一、电梯陈述（三个版本，按场景切换）

### 30 秒版（破冰用）

> RAP 是一个覆盖科研全流程的 AI Agent 系统，从提出科学问题、检索文献、设计实验、评估结果到撰写投稿论文，8 个阶段由专业 Agent 协作完成。核心差异化在于：人在回路（HIL）的审阅机制保证学术严谨性，冷启动 RAG 让用户无需预建知识库即可开箱即用。

### 1 分钟版（自我介绍衔接）

> 我做了一个叫 Research Auto-Pilot 的项目，定位是"科研工作者的 AI 协作者"。传统科研工具是"单点提效"——文献管理用 Zotero、写代码用 Jupyter、写作用 Overleaf，但跨阶段的知识传递和流程编排全靠人。RAP 用多智能体架构把 8 个阶段串成一条流水线，每个阶段由专门的 Agent 负责，并在方案确认、实验设计、结果评估、论文定稿 4 个关键节点设置人工审阅中断点。技术上用 LangGraph 做状态机编排，RAG 作为独立服务层支持混合检索，LLM 按任务路由到不同档位的国产模型以平衡效果和成本。

## 二、问题定义与用户洞察

### 2.1 痛点（先讲"为什么做"）

| 用户痛点 | 现状 | RAP 的解法 |
|---------|------|-----------|
| 跨阶段知识断层 | 文献读完了，到设计实验时还得重新翻笔记 | Agent 间状态传递，文献 Agent 的输出直接喂给实验设计 Agent |
| 重复性劳动占用创造时间 | 格式排版、引用核对、图表生成 | 写作 Agent 自动校验 refs.bib，图表 Agent 自动出图 |
| 低资源场景下科研门槛高 | 不是人人有 GPU 集群 | LoRA + RAG 组合 + 国产经济模型路由 |
| 工具割裂 | 10 个工具切换，上下文丢失 | 一个工作台统一编排 |

### 2.2 目标用户（分三层，体现 PM 的市场分层思维）

- **核心用户**：高校硕博研究生（科研生产力主力，痛点最强、付费意愿培养期）
- **扩展用户**：企业 R&D 研究员（预算充足，B 端化重点）
- **愿景用户**：独立研究者 / 跨学科研究者（长尾，但能讲"降低科研门槛"的故事）

### 2.3 用户画像（面试时讲一个具象故事）

> 小李是医学 NLP 方向的硕士，开题后要花 3 周读 80 篇文献、2 周设计实验、1 个月跑实验、2 周写论文。其中真正需要"科研判断力"的环节不到 40%，剩下都是检索、整理、排版、核对。RAP 要替代的就是那 60%。

## 三、产品定位与差异化

### 3.1 一句话定位

> **"不替代科研判断，而接管科研流程中的确定性劳动，让人专注在创造性环节。"**

这句话很关键——它划清了 AI 与人的边界，也是回答"学术伦理"质疑的底气。

### 3.2 竞品对比（PM 必备）

| 维度 | ChatGPT/Claude | Elicit/Semantic Scholar | AutoGen/CrewAI | **RAP** |
|------|---------------|------------------------|----------------|---------|
| 流程编排 | 单轮对话 | 文献检索单点 | 通用 Agent 框架 | **科研全流程编排** |
| 领域知识 | 通用 | 文献元数据 | 无 | **RAG 科研知识库** |
| 人工审阅 | 无 | 无 | 需自建 | **4 个 HIL 中断点** |
| 产物完整性 | 文本 | 检索结果 | 代码 | **论文+实验+图表+投稿** |
| 学术合规 | 弱 | 中 | 无 | **引用校验 + 可追溯** |

### 3.3 差异化护城河（讲清"为什么别人抄不走"）

1. **流程编排的领域知识**——8 阶段 + 4 HIL 的设计是科研方法论沉淀，不是套壳
2. **RAG 冷启动策略**——70% 开放学术 API + 懒加载，零冷启动成本
3. **引用可追溯 schema**——`{paper_id, section, page}` 强校验，解决 AI 写作的幻觉硬伤

## 四、产品架构与核心功能

### 4.1 系统架构（三层，PM 要能画出来）

```
前端 React（:5173）  ←→  后端 Node.js/Express（:3001）  ←→  LLM 服务 Python/FastAPI（:8000）
       │                        │                              │
   工作台/HIL/编辑器      Orchestrator + 项目管理         LangGraph 状态机 + Agent + RAG
```

### 4.2 核心 Agent 流水线（8 阶段）

```
literature → design → experiment → evaluation → discussion → writing → figure → submission
   ↑           ↑          ↑            ↑                                    ↑
 [HIL-1]    [HIL-2]    (自动)        [HIL-3]                             [HIL-4]
```

4 个 HIL 中断点的位置是**产品决策**，不是技术决策——讲清楚为什么这 4 个点要人审：

- **HIL-1 文献确认**：方向错了后面全错，成本最低的拦截点
- **HIL-2 方案确认**：实验设计决定资源投入，必须人批
- **HIL-3 结果评估**：科研结论的诚实性底线
- **HIL-4 论文定稿**：投稿即公开发表，最后把关

### 4.3 功能模块（对应 UI 8 页面）

仪表盘 / 项目工作台 / 人审中断点 / 文献库 / 实验监控 / 论文编辑器 / 版本管理 / 系统配置 + 账号中心

## 五、AI 技术理解（PM 视角，不背代码）

这是 AI 产品经理面试的**核心考察点**。要讲"懂原理、懂边界、懂取舍"。

### 5.1 为什么用多 Agent 而不是单 Prompt

- 单 Prompt 上下文爆炸，8 阶段塞不下
- 各阶段任务异构（检索 vs 代码 vs 写作），需要不同系统提示和工具
- Agent 间可并行/可中断，状态机可控

### 5.2 为什么用 LangGraph

- 显式状态机，支持 HIL 中断（`interrupt`）和回滚
- 状态可持久化，刷新不丢
- 比纯 ReAct 更可控，比 AutoGen 更轻

### 5.3 RAG 为什么做成独立服务层（这是项目硬约束）

- 解耦：Agent 不关心检索实现，只调接口
- 复用：多 Agent 共享同一知识库
- 可演进：可独立升级检索策略（向量→混合→重排）

### 5.4 LLM 四档路由（讲成本意识）

| 档位 | 模型 | 场景 | 为什么 |
|------|------|------|--------|
| 强推理 | DeepSeek-R1 | 方案设计/讨论分析 | 需要深度推理 |
| 经济 | DeepSeek-V3 | 写作/总结 | 性价比 |
| 长文本 | Kimi 200K | 长文献处理 | 上下文窗口 |
| Embedding | bge-m3 | RAG 向量化 | 中文效果好 |

**PM 话术**："这不是技术选型，是单位任务的成本优化——把贵模型用在推理瓶颈上。"

### 5.5 RAG 冷启动（懒加载）

- 痛点：预建知识库冷启动成本高、维护贵
- 解法：用户提问时实时拉 arXiv/S2/OpenAlex，按章节结构化切片
- 数据配比：70% 开放 API / 20% 用户种子 / 10% 系统回流

### 5.6 幻觉控制（AI 产品必问）

- 写作 Agent 强制校验引用 against `refs.bib`，schema `{paper_id, section, page}`
- RAG 混合检索（向量 + BM25 + metadata）降低检索幻觉
- HIL 在关键节点兜底

## 六、关键产品决策（讲取舍，PM 的含金量所在）

面试官最爱问"为什么这样设计而不是那样"。准备 3 个决策故事：

### 决策 1：HIL 而非全自动

- 取舍：全自动更快但不可控；HIL 慢但可信
- 选择 HIL 的理由：科研场景"可信 > 效率"，一次学术事故毁掉产品口碑
- 量化：4 个中断点增加约 30% 流程时间，换来 100% 人工把关

### 决策 2：RAG 冷启动而非预建库

- 取舍：预建库检索快但覆盖窄、维护贵；冷启动慢但零成本、全覆盖
- 理由：长尾学科无法预建，懒加载符合"通用学科"定位
- 代价：首问延迟高，用 streaming + 缓存缓解

### 决策 3：国产模型路由而非 GPT-4

- 取舍：GPT-4 效果稳但贵且合规风险；国产模型需路由但成本低、合规
- 理由：国内高校/企业场景的合规和成本约束
- 代价：需自建路由逻辑，效果波动靠 RAG 补偿

## 九、风险与边界（最容易被 challenge，要主动讲）

### 9.1 学术伦理（最大风险）

- **质疑**："AI 写论文算学术不端吗？"
- **回应**：RAP 定位是"协作者"非"代笔"，HIL 保证人的判断在场，所有 AI 产物可追溯、可标注。类比 Grammarly/DeepL 是写作工具，RAP 是流程工具。
- **产品设计**：投稿时自动生成 AI 使用声明

### 9.2 幻觉风险

- 引用幻觉 → refs.bib 强校验
- 实验数据幻觉 → 实验代码独立可运行 + dry-run + 日志持久化

### 9.3 能力边界（诚实讲做不到什么）

- 不能替代原创性科学洞察（HIL-1 方向仍由人定）
- 不能保证实验可复现（依赖用户执行环境）
- 长尾学科 RAG 覆盖不足

## 十、面试高频问题与应对（STAR 法则）

**Q1：这个项目和 ChatGPT 有什么区别？**
> ChatGPT 是通用对话工具，单轮无状态；RAP 是领域 Agent 系统，8 阶段状态机 + HIL + RAG + 产物完整性。它解决的是"流程"而非"问答"。

**Q2：你怎么验证产品价值？**
> 北极星是端到端转化率。Pilot 阶段用 5 个真实课题，对比"用 RAP vs 不用 RAP"的投稿周期和 HIL 一次通过率。先验证"降耗时"，再验证"提质量"。

**Q3：Agent 失败了怎么办？**
> 三层兜底：单 Agent 内重试 → LangGraph 回滚到上一阶段 → HIL 人工介入。版本管理在 EXPERIMENT→EVALUATE 切换时自动快照到 `_versions/vN/`，可回滚。

**Q4：成本怎么控？**
> LLM 四档路由 + RAG 缓存 + 经济模型兜底。单项目 token 成本作为核心反向指标监控，超阈值自动降级模型档位。

**Q5：为什么选 LangGraph 不选 AutoGen？**
> LangGraph 显式状态机更适合有明确阶段和 HIL 的流程；AutoGen 偏多 Agent 对话，状态管理弱。科研流程是"确定性阶段 + 不确定性执行"，LangGraph 的可控性更匹配。

**Q6：你作为 PM 的最大取舍是什么？**
> 砍掉"全自动投稿"功能。技术上能做，但学术伦理风险太高，会毁掉产品在核心用户群的信任。这是短期功能换长期信任的决策。

# 第二部分：Agent 技术模式深度分析（ReAct vs Plan vs Workflow）

> 这一章回答一个高频技术追问："你这个项目用的是 ReAct 还是 Plan？"
> 答案：**都不是，是预定义工作流（Workflow / 状态机）**。这点在 AI 产品经理面试里非常容易被深挖，必须讲清楚。

## 结论

本项目用的是 **预定义工作流（Workflow / 状态机）**，不是 ReAct，也不是 Plan-and-Execute。

## 三种模式对比

| 模式 | 核心机制 | 谁决定下一步 | 适用场景 | 代价 |
|------|---------|------------|---------|------|
| **ReAct** | Thought→Action→Observation 循环，LLM 自主反思 | LLM 每步自主决定 | 开放式探索、工具组合不可预知 | 不可控、易跑偏、token 消耗大 |
| **Plan-and-Execute** | 先 LLM 生成完整计划，再逐步执行 | LLM 一次性规划 | 任务结构已知但步骤动态 | 规划错则全错，重规划贵 |
| **Workflow（本项目）** | 人工预设节点 + 边，代码编排 | 开发者预先定义 | 流程确定、阶段清晰的领域 | 灵活性低，新增阶段要改图 |

## 本项目的实际实现

### 整体编排层

`llm_service/src/agents/orchestrator.py` 用 LangGraph 的 `StateGraph`，8 个节点用 `add_edge` 串成**固定线性流水线**：

```python
workflow.add_edge("literature_review", "design")
workflow.add_edge("design", "experiment")
# ... 固定顺序，无 LLM 决策分支
```

边是写死的，**LLM 不参与"下一步去哪"的决策**。

### 单节点内部

`llm_service/src/agents/literature_agent.py` 也不是 ReAct 循环，而是**硬编码的工具调用序列 + 单次 LLM 总结**：

```python
# 步骤1: RAG 检索
rag_docs = await get_rag_service().query(question, top_k=5)
# 步骤2: arXiv
arxiv_docs = await search_arxiv(question, limit=5)
# 步骤3: S2
s2_docs = await search_s2(question, limit=5)
# 步骤4: 合并
# 步骤5: LLM 总结（只调一次，不反思）
summary = await llm_generate(prompt, tier="strong")
```

LLM 只在最后做一次总结，**不决定调哪个工具、不反思结果好不好、不重试**——这是典型的 workflow 节点，不是 ReAct agent。

## 为什么选 Workflow 而不是 ReAct/Plan（面试加分点）

### 理由 1：科研流程是确定性的，不需要 LLM 自主探索

> 科研方法论本身就是 literature→design→experiment→... 的固定流程。阶段顺序是领域知识沉淀，不该交给 LLM 临场决定。用 ReAct 等于把"科研怎么做"这个领域专家才知道的事，丢给一个可能跑偏的 LLM。

### 理由 2：可控性和可审计性

> 科研场景要求每一步可追溯、可回滚。ReAct 的自主循环轨迹不可预测，无法保证"到 design 前一定停下来等人审"。Workflow + `interrupt_before` 能精确卡住 4 个 HIL 点。

### 理由 3：成本和延迟

> ReAct 每步都要 LLM 决策（多一轮推理），8 阶段下来 token 翻倍。Workflow 节点内的工具调用是代码直连，只在需要生成内容时调一次 LLM。

### 理由 4：Plan-and-Execute 的规划在这里是冗余

> 流程已知，没必要先花一次 LLM 调用去"规划"一遍固定的 8 步。

## 取舍的代价（讲取舍才显成熟）

- **灵活性低**：长尾学科若流程不同（比如纯理论学科不需要 experiment），要改图。目前用"占位 + 跳过日志"缓解，但不是真分支。
- **节点内工具固定**：literature_agent 硬编码了 arXiv+S2，无法根据问题动态选 PubMed。要升级成"节点内 ReAct"才能动态选工具——这是后续迭代方向。

## 演进路径（体现产品迭代思维）

```
当前：纯 Workflow（固定图 + 固定节点内工具）
  ↓ 下一步
节点内 ReAct：图结构固定，但节点内 LLM 自主选工具
              （如 literature 节点内 LLM 决定调哪几个 API）
  ↓ 再下一步
混合 Plan：流程入口先让 LLM 规划"这个学科要不要跳过 experiment"，生成子图
```

## 面试一句话话术

> "我们用的是**预定义工作流 + LangGraph 状态机**，不是 ReAct。因为科研流程的阶段顺序是领域知识，不该让 LLM 临场决定；但节点内部未来会引入 ReAct 让 LLM 自主选检索工具。这是**流程确定性交给代码、内容生成不确定性交给 LLM**的分层设计。"

这句"流程确定交给代码、内容生成交给 LLM"是核心判断力体现，面试官会记住。

## 附：快速回顾卡（面试前 5 分钟扫一遍）

- **定位**：科研全流程 AI 协作者，8 阶段 + 4 HIL
- **架构**：React + Node + Python，LangGraph 状态机，RAG 独立服务层
- **技术模式**：Workflow（非 ReAct 非 Plan），"流程交代码、内容交 LLM"
- **路由**：DeepSeek-R1/V3 + Kimi 200K + bge-m3 四档
- **冷启动**：70% API / 20% 种子 / 10% 回流，懒加载
- **幻觉控制**：refs.bib 强校验 `{paper_id, section, page}` + 混合检索 + HIL
- **北极星**：端到端转化率（投稿/启动）
- **3 个决策**：HIL 非全自动 / 冷启动非预建 / 国产路由非 GPT-4
- **最大风险**：学术伦理 → 定位协作者 + AI 使用声明
- **一句话**：不替代科研判断，接管确定性劳动
